ENGENHARIA DE REQUISITOS 

->É o processo de adequação dos projetos de engenharia a um conjunto de requisitos
básico de software / importante para criar resultados precisos em engenharia de 
software

.o que o cliente quer?

.o que o usuario precisa para usar o sistema?

.qual será o impacto do software nos usuarios?

O QUE SE FAZ 

->Analiza-se um conjunto de dados referentes as metas e objetivos do software 

->Como ele funcionará e quais são as qualidades das propriedades que ele deve
ter para fornecer os resultados necessários.

OS REQUISITOS INCLUEM 

->Solicitação de requisitos , onde uma empresa de software obtém os requisitos
de um cliente

->Analise de requisitos

->Especificação de requisitos 

->Verificação de requisitos, onde os engenheiros confirmam que os requisitos 
são precisos 

->Gerenciamento de requisitos , que combina processos com seus requisitos

COMO ACONTECE 

->Uma parte importante da engenharia tem a ver com as partes interessadas ou partes 
envolvidas no processo

->Muitos estágios da engenharia de requisitos acontecem durante as comunicações
entre o cliente e a empresa de software

->Engenharia de requisitos tenta unir essa divisão entre o que o cliente e o
que os desenvolvedores estão pensando 

->E criar uma estrutura solida e consistente para a construçao  de produtos de 
software sofisticados

VARIAS ETAPAS PARA DESCOBERTA

CONCEPÇÃO: Em que a natureza do escopo do sistema são definidos

ELICITAÇÃO: Em que os requisitos para o software são inicialmente reunidos

ELABORAÇÃO: Na qual os requisitos reunidos são refinados

NEGOCIAÇÃO; Na qual as prioridades de cada requisito são determinadas, os requisitos
essenciais são anotados e, mais importante: os conflitos entre os requisitos
são resolvidos

ESPECIFICAÇÃO: Na qual os requisitos são reunidos em um unico produto, sendo o 
resultado da engenharia de requisitos

VALIDAÇÃO: Na qual a qualidade dos requisitos (ou seja, eles são inequívocos, 
consistentes , completos, etc.)

->E a interpretação do desenvolvedor deles , são validados

GERENCIAMENTO; No qual as mudanças que os requisitos devem sofrer durante a vida
do projeto são gerenciadas.

PAPEIS DE ENGENHARIA DE REQUISITOS

->Formatar a base da arquitetura do sistema e das atividades de design

->Formar a base das atividades de integração e verificação do sistema

->Atuar como referencia para a validação e aceitação das partes interessadas

->Fornecer um meio de comunicação entre os vários funcionarios técnicos que 
interagem durante o projeto

REQUISITOS FUNCIONAIS -> Descreve qualitativamente as funções ou tarefas do sistema
a serem executadas em operação

REQUISITOS DE DESEMPENHO -> Define a extensão , ou quão bem, e sob quais 
condições uma função ou tarefa deve ser executada 

->estes são os requisitos quantitativos de desempenho do sistema e são verificaveis
individualmente

->Pode haver mais de um requisito de desempenho associado a uma única função,
requisito funcional ou tarefa

REQUISITOS DE USABILIDADE -> Definir a qualidade do uso do sistema (por exemplo
, eficácia mensurável , eficiência e critérios de satisfação)

REQUISITOS DE INTERFACE ->Defina como o  sistema é necessário para interagir 
ou trocar material, energia ou informações com sistemas externos (interface 
externa) ou como os elementos do sistema dentro do sistema

->Inlcuindo elementos humanos , interagem uns com os outros (interface interna)

->Os requisitos de interface incluem conexões fisicas (interfaces fisicas) com
sistemas externos ou elementos internos do sistema que suportam interações ou 
trocas

REQUISITOS OPERACIONAIS -> Defina as condições operacionais ou propriedades necessárias
para o sistema operar ou existir 

->Este tipo de requisito inclui : fatores humanos , ergonomia , disponibilidade
, facilidade de manutenção, confiabilidade e segurança 

REQUISITOS DE MODOS E / OU ESTADOS -> Defina os vários modos operacionais do 
sistema em uso e os eventos que conduzem a transições de modos

REQUISITOS DE ADAPTIBILIDADE -> Defina extensão , crescimento ou escalabilidade em potencial
durante a vida útil do sistema 

RESTRIÇÕES FÍSICAS -> Defina as restrições de peso, volume e dimensão aplicáveis
aos elementos do sistema que compõem o sistema

RESTRIÇÕES DE DESIGN ->Defina os limites das operaçoes disponiveis para um
projetista de uma solução , impondo limites e limites imutáveis 

-> EX , o sistema deve incorporar um legado ou um elemento do sistema fornecido
ou determinados dados devem ser mantidos em  um repositório on-line).

CONDIÇÕES AMBIENTAIS ->Defina as condições ambientais a serem encontradas pelo
sistema em seus diferentes modos operacionais

->Isto deve abordar o ambiente natural (tempo, chuva, vento, temperatura, fauna
,radiação, sal , poeira , etc)

->Efeitos ambientais induzidos e / ou auto induzidos (movimento , choque, ruido
, eletromagnetismo , térmico , etc)

->E ameaças ao ambiente social (legais ,politicos , economicos , sociais , 
comerciais , etc.)

REQUISITOS LOGISTICOS -> Defina as condições logisticas necessárias para a utilização 
continua do sistema

->Esses requisitos incluem a manutenção (fornecimento de instalações , suporte
de nivel , pessoal de suporte , peças sobressalentes, treinamentos , documentação
, etc)

POLITICAS E REGULAMENTOS -> Definir politicas organizacionais relevantes ou 
aplicaveis ou requisitos regulatorios  que possam afetar a operação ou o 
desempenho do sistema (politicas trabalhistas , relatórios para a agonia 
regulatória , critérios de saúde ou segurança)

RESTRIÇÕES DE CUSTO E CRONOGRAMA ->Definir , o custo de um unico exemplar do
sistema , a data de entrega esperada do primeiro exemplar
